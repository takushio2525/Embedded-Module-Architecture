\section{概要}

\subsection{プロジェクト名}
Embedded Module Architecture — Arduino対応 組み込みモジュール設計パターン

\subsection{コンセプト}
本プロジェクトは、Arduinoフレームワーク対応マイコン全般で使える\textbf{Moduleインターフェース設計パターン}のリファレンス実装である。
コア設計（IModule・ModuleTimer）はESP32に限らず、STM32やAVR等のArduino互換ボードにもそのまま適用できる。

\subsection{リポジトリ構成}

本リポジトリには2つのプロジェクトが含まれる。各プロジェクトは完全に自己完結しており、フォルダ単体でPlatformIOプロジェクトとして使用できる。

\begin{tabularx}{\textwidth}{l X}
\toprule
\textbf{ディレクトリ} & \textbf{内容} \\
\midrule
\texttt{sample/} & 設計パターンの構造説明用サンプル（ESP32-S3 N16R8 CAM題材、動作未確認） \\
\texttt{verified/} & 実機で動作確認済みのプロジェクト \\
\bottomrule
\end{tabularx}

\begin{cautionbox}[sample/ のコードについて]
\texttt{sample/} に実装されているモジュール群のコードは、あくまで\textbf{アーキテクチャの設計パターンを具体的に示すためのサンプル実装}である。
実機での動作検証は行っていない。
実プロジェクトへの適用時は、ハードウェア固有の動作・タイミング・電気的仕様を考慮したうえで十分なテストを実施すること。
動作確認済みのコードは \texttt{verified/} を参照すること。
\end{cautionbox}

\subsection{設計思想}

\begin{funcblock}{基本理念：どんなプロジェクトにも流用できる汎用的なコアクラス}
IModule・ModuleTimerは\textbf{プロジェクトに一切依存しない}汎用コンポーネントとして設計する。
\texttt{SystemData}は前方宣言のみで参照するため、プロジェクト固有のヘッダーに依存しない。
コアクラスをコピーするだけで、どんな組み込みプロジェクトにもそのまま流用可能である。
\end{funcblock}

\begin{funcblock}{基本理念：各モジュールはそれ単体で動作する道具}
モジュールは他のモジュールの存在を知らず、\textbf{自身のSystemDataサブ構造体のみ}を参照する。
入力モジュールは「ハードウェアから値を読んでSystemDataに書く」だけ、
出力モジュールは「SystemDataの値に従ってハードウェアを制御する」だけの単機能な道具である。
「何をどう制御するか」というロジックはモジュールの外（ロジックフェーズ）で行う。
\end{funcblock}

\begin{funcblock}{基本理念：すべてのモジュールは再利用可能な部品}
コアクラス（IModule・ModuleTimer）だけでなく、\textbf{IModuleを継承したすべての子クラス（各モジュール実装）も、
別のプロジェクト・別の開発者にそのまま流用できる}ことを前提として設計する。
一度作成したモジュールはプロジェクト間で共有し、誰でも同じように使える「共有部品」として扱う。\\[6pt]
この前提を守るためのルール：
\begin{itemize}
    \item モジュール内にプロジェクト固有のロジックを記述しない（入力は読み取るだけ、出力はSystemDataに従うだけ）
    \item ハードウェア依存の設定（ピンアサイン等）はConfig構造体で外部から注入する
    \item モジュールが参照するSystemDataサブ構造体は、モジュール自身のヘッダ内で定義し、モジュールと一緒に持ち運べるようにする
    \item モジュールが他のモジュールのサブ構造体を直接参照しない
\end{itemize}
\end{funcblock}

本アーキテクチャの根幹となる5つの要素：

\begin{enumerate}
    \item \textbf{IModuleインターフェース} — \texttt{SystemData}の前方宣言のみを使用し、プロジェクト固有の型に依存せず、
    全モジュールが\texttt{init()}と\texttt{update(SystemData\& data)}を実装する統一的なライフサイクル管理
    \item \textbf{ModuleTimerクラス} — \texttt{delay()}を一切使用しないノンブロッキングなタイマー制御
    \item \textbf{SystemDataパターン} — プロジェクト固有の共有状態をサブ構造体で一元管理し、モジュール間のデータ連携を実現
    \item \textbf{モジュールConfigパターン} — 各モジュールのピンアサインや設定をConfig構造体として定義し、
    コンストラクタで受け取ることでハードウェア構成を明示的かつ一元的に管理
    \item \textbf{3フェーズ実行モデル} — ループ内の処理を「入力→ロジック→出力」の3段階に分離し、
    モジュールの独立性を保証する実行手順
    \item \textbf{通信バスパターン} — SPI・I2C等の共有バスインスタンスを\texttt{main.cpp}で一元管理し、
    ポインタでモジュールに渡すことでバスの競合を防止
\end{enumerate}

この設計により、以下を実現する：
\begin{itemize}
    \item \textbf{コアクラスの完全な汎用性}（プロジェクト非依存）
    \item \textbf{全モジュールのプロジェクト横断的な再利用性}（子クラスも含め、作ったモジュールをそのまま別プロジェクトで共有可能）
    \item \textbf{モジュールの独立性}（各モジュールは単体で動作する道具）
    \item 入出力両方を持つデバイスもモジュール分離により独立性を維持
    \item モジュールの追加・削除が容易
    \item 処理順序が決定論的（入力→ロジック→出力の3フェーズ）
    \item 複数回出力や処理の重複が発生しない
    \item ゲームループ（ECS）に類似した、組み込みに適した実行モデル
    \item 参照渡し（\texttt{\&}）によるnullポインタ問題の排除
    \item \textbf{ピンアサインの一元管理}（Config構造体により回路設計者がピン定義ファイルだけで把握可能）
    \item \textbf{通信バスの安全な共有}（SPI・I2Cバスを\texttt{main.cpp}で一元管理し、ポインタで渡すことで競合を防止）
\end{itemize}
