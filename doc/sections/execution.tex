\section{実行モデル詳細}

\subsection{ライフサイクル}

本アーキテクチャのモジュールは、以下のライフサイクルに従う：

\begin{enumerate}
    \item \textbf{インスタンス生成}（グローバルスコープ） \\
    モジュールのコンストラクタが呼ばれる。\texttt{ProjectConfig.h}で定義したConfig定数を渡し、
    メンバ変数\texttt{\_config}にコピーされる。ModuleTimerの\texttt{\_startTime = 0}で初期化される。
    この時点では\texttt{millis()}は信頼できないため、タイマーの実質的な初期化は行わない。

    \item \textbf{バス初期化}（\texttt{setup()}内の最初） \\
    SPI・I2C等の通信バスを使用する場合、\texttt{SPI.begin()}や\texttt{Wire.begin()}等のバス初期化を
    全モジュールの\texttt{init()}より前に実行する。
    バスインスタンスは\texttt{main.cpp}のグローバルスコープで生成し、コンストラクタ引数でモジュールに渡す。

    \item \textbf{init()呼び出し}（\texttt{setup()}内） \\
    ハードウェアの初期化、ピンモード設定、ModuleTimerの\texttt{setTime()}呼び出しを行う。
    失敗時は\texttt{false}を返し、呼び出し側でエラーハンドリングを行う。

    \item \textbf{updateInput()/updateOutput()ループ}（\texttt{loop()}内） \\
    毎ループ呼び出される。\texttt{SystemData}の参照を受け取り、ModuleTimerの\texttt{getNowTime()}で経過時間を確認し、
    必要なタイミングでのみ実処理を実行する。入力モジュールは\texttt{updateInput()}のみ、
    出力モジュールは\texttt{updateOutput()}のみをオーバーライドする。
    入出力両方を持つモジュールは両方をオーバーライドする。

    \item \textbf{deinit()呼び出し}（必要時） \\
    スリープ突入前やモジュール停止時に呼び出す。ペリフェラルの解放、GPIO状態のリセット等のクリーンアップ処理を行う。
    デフォルトは空実装であり、リソース解放が必要なモジュールのみオーバーライドする。
\end{enumerate}

\subsection{ループ実行手順（3フェーズモデル）}

\texttt{loop()}内の処理は、以下の3フェーズに分けて実行する。
この順序を守ることで、モジュール間の暗黙的な依存関係を排除し、各モジュールの独立性を保つ。

\begin{enumerate}
    \item \textbf{入力フェーズ} — 全\texttt{inputModules[]}の\texttt{updateInput()}を実行。
    ハードウェアから値を読み取り、自身の\texttt{SystemData}サブ構造体に書き込む。
    \item \textbf{ロジックフェーズ} — 入力系の\texttt{SystemData}を読み取り、
    出力系の\texttt{SystemData}を書き換える制御ロジックを実行する。
    \item \textbf{出力フェーズ} — 全\texttt{outputModules[]}の\texttt{updateOutput()}を実行。
    自身の\texttt{SystemData}サブ構造体の値に従ってハードウェアを制御する。
\end{enumerate}

\begin{notebox}[3フェーズの原則]
各モジュールは「それ単体で動作する道具」である。\\
入力モジュールは「値を読んでSystemDataに書く」だけ、出力モジュールは「SystemDataの値に従って動作する」だけ。\\
モジュールの\texttt{updateInput()}/\texttt{updateOutput()}内で他モジュールのSystemDataを参照してはならない。\\
「何をどう制御するか」の判断はロジックフェーズの責務である。
\end{notebox}

\subsubsection{ロジックフェーズの実装}

ロジックフェーズの具体的な実装方法は開発者に委ねるが、標準的には関数として実装する。

\begin{lstlisting}[style=cppstyle, caption={ロジックフェーズの標準的な実装}]
// main.cpp
void applyPattern(SystemData& data) {
    // 入力SystemDataを読み、出力SystemDataを書き換える
    // 例：スイッチが押されたらLEDを点灯
    data.led.state = data.sw.isPressed;

    // 例：温度が30度を超えたらモーターを回す
    if (data.temp.isValid && data.temp.temperature > 30.0) {
        data.motor.speed = 255;
    } else {
        data.motor.speed = 0;
    }
}

void loop() {
    // 1. 入力フェーズ
    for (int i = 0; i < INPUT_COUNT; i++) {
        if (inputModules[i]->enabled) {
            inputModules[i]->updateInput(systemData);
        }
    }
    // 2. ロジックフェーズ
    applyPattern(systemData);
    // 3. 出力フェーズ
    for (int i = 0; i < OUTPUT_COUNT; i++) {
        if (outputModules[i]->enabled) {
            outputModules[i]->updateOutput(systemData);
        }
    }
}
\end{lstlisting}

\subsubsection{入出力分離ルール}

1つのデバイスが入力と出力の両方の機能を持つ場合、\textbf{1つのモジュールで\texttt{updateInput()}と\texttt{updateOutput()}の両方をオーバーライド}し、
\texttt{inputModules[]}と\texttt{outputModules[]}の\textbf{両方の配列に含める}。

\begin{tabularx}{\textwidth}{l l l}
\toprule
\textbf{デバイス} & \textbf{updateInput()の処理} & \textbf{updateOutput()の処理} \\
\midrule
BLE通信 & データ受信（内部バッファ→SystemData） & データ送信（SystemData→BLE） \\
LCD＋タッチ & タッチ座標取得 & 画面描画 \\
\bottomrule
\end{tabularx}

\begin{lstlisting}[style=cppstyle, caption={入出力モジュールの配列配置}]
// 入出力モジュールは両方の配列に含める
IModule* inputModules[] = {
    &bleModule,    // BLE受信を先に反映
    &touchModule,
    &mpu6500Module,
};

IModule* outputModules[] = {
    &ledModule,
    &tftModule,
    &bleModule,    // 全データ確定後にBLE送信
};
\end{lstlisting}

メソッド分離により、入力処理と出力処理が構造的に分かれるため、1つのモジュールが両方の責務を持っても安全に動作する。
\texttt{setup()}では全ユニークモジュールを1回だけ\texttt{init()}する（両配列に含まれるモジュールの重複initを避ける）。

\subsubsection{アンチパターン}

出力モジュール内で入力系の\texttt{SystemData}を読み取ってロジックを書く行為は\textbf{禁止}とする。

\begin{lstlisting}[style=cppstyle, caption={アンチパターン：出力モジュール内でロジックを持つ（禁止）}]
// NG: LEDモジュール内で温度を判断している
void LedModule::updateOutput(SystemData& data) {
    // 他モジュールのSystemDataを読んではならない
    if (data.temp.temperature > 30.0) {
        data.led.state = true;
    }
    digitalWrite(_config.ledPin, data.led.state);
}
\end{lstlisting}

\begin{lstlisting}[style=cppstyle, caption={正しい実装：出力モジュールは自身のSystemDataのみ参照}]
// OK: LEDモジュールは自身のデータだけを見る
void LedModule::updateOutput(SystemData& data) {
    // 自身のサブ構造体の値に従って動作するだけ
    digitalWrite(_config.ledPin, data.led.state);
}

// ロジックは別の場所（applyPattern等）で行う
void applyPattern(SystemData& data) {
    data.led.state = (data.temp.temperature > 30.0);
}
\end{lstlisting}

\subsection{タイミング制御}

\texttt{delay()}は一切使用しない。すべてのタイミング制御はModuleTimerで行う。

\begin{tabularx}{\textwidth}{l l X}
\toprule
\textbf{パターン} & \textbf{方法} & \textbf{コード例} \\
\midrule
周期実行 &
\texttt{getNowTime() >= interval}で判定し、\texttt{setTime()}でリセット &
1秒ごとのセンサー読み取り \\

\subsection{モジュール間データフロー}

3フェーズモデルにおける\texttt{SystemData}を介したデータフローの例：

\begin{lstlisting}[style=cppstyle, caption={3フェーズモデルによるデータフロー}]
// --- 入力フェーズ ---
// 温度センサーモジュール：ハードウェアから値を読み、自身のデータに書く
void TempSensorModule::updateInput(SystemData& data) {
    if (_readTimer.getNowTime() >= _config.interval) {
        _readTimer.setTime();
        data.temp.temperature = analogRead(_config.tempPin);
        data.temp.isValid = true;
    }
}

// --- ロジックフェーズ（main.cpp内の関数等） ---
void applyPattern(SystemData& data) {
    // 温度に応じてLEDを制御する判断をここで行う
    if (data.temp.isValid && data.temp.temperature > 30.0) {
        data.led.state = true;
    } else {
        data.led.state = false;
    }
}

// --- 出力フェーズ ---
// LEDモジュール：自身のデータに従って出力するだけ
void LedModule::updateOutput(SystemData& data) {
    digitalWrite(_config.ledPin, data.led.state);
}
\end{lstlisting}

\subsection{割り込み実行パターン}

リアルタイム性が求められるプロジェクトや、カメラ等の重い処理を含むプロジェクトでは、
3フェーズループをタイマー割り込みで周期実行し、メインループは重い処理に専念する構成をとる。

\begin{lstlisting}[style=cppstyle, caption={タイマー割り込みによる3フェーズ実行}]
// main.cpp
hw_timer_t* moduleTimer = NULL;
volatile bool updateFlag = false;

// タイマー割り込みハンドラ
void IRAM_ATTR onModuleTimer() {
    updateFlag = true;
}

void setup() {
    Serial.begin(115200);

    // モジュール初期化（省略）

    // タイマー割り込み設定（10ms周期）
    moduleTimer = timerBegin(0, 80, true);  // 80分周 = 1MHz
    timerAttachInterrupt(moduleTimer, &onModuleTimer, true);
    timerAlarmWrite(moduleTimer, 10000, true);  // 10ms
    timerAlarmEnable(moduleTimer);
}

void loop() {
    // メインループ：重い処理（カメラ取得等）を実行
    cameraModule.capture();  // 時間のかかる処理

    // 割り込みフラグで軽量モジュール更新
    if (updateFlag) {
        updateFlag = false;
        // 1. 入力フェーズ
        for (int i = 0; i < INPUT_COUNT; i++) {
            if (inputModules[i]->enabled) {
                inputModules[i]->updateInput(systemData);
            }
        }
        // 2. ロジックフェーズ
        applyPattern(systemData);
        // 3. 出力フェーズ
        for (int i = 0; i < OUTPUT_COUNT; i++) {
            if (outputModules[i]->enabled) {
                outputModules[i]->updateOutput(systemData);
            }
        }
    }
}
\end{lstlisting}

\begin{notebox}[割り込み実行パターンの使い分け]
シンプルなプロジェクトでは通常の\texttt{loop()}内での3フェーズ実行で十分である。\\
割り込み実行パターンは、重い処理（カメラ・画像処理・FFT等）と
軽量な周期処理（センサー読み取り・LED制御等）を共存させる必要がある場合に使用する。
\end{notebox}

\subsection{外部データアクセスパターン（スレッドセーフティ）}

3フェーズループの外（割り込み・別タスク・BLEコールバック等）から\texttt{SystemData}に直接アクセスしてはならない。
外部からのデータは、\textbf{モジュール内部バッファ + \texttt{volatile}フラグ方式}で受け渡す。
割り込みでもマルチタスクでも同じルールを適用する。

\begin{lstlisting}[style=cppstyle, caption={外部データアクセスパターン（volatile フラグ方式）}]
class BleModule : public IModule {
private:
    BleConfig _config;
    // 内部バッファ（BLEコールバックが書き込む）
    uint8_t _rxBuffer[128];
    uint8_t _rxLength = 0;
    volatile bool _dataReceived = false;  // volatileフラグ

public:
    BleModule(const BleConfig& config) : _config(config) {}

    // BLEコールバックから呼ばれる（別タスクで実行される）
    void onReceive(uint8_t* data, uint8_t length) {
        memcpy(_rxBuffer, data, length);  // バッファに書き込み
        _rxLength = length;
        _dataReceived = true;             // フラグを立てる
    }

    // updateInput()は入力フェーズで呼ばれる
    void updateInput(SystemData& sysData) override {
        if (_dataReceived) {
            // フラグが立っていたらバッファからSystemDataへコピー
            memcpy(sysData.ble.rxData, _rxBuffer, _rxLength);
            sysData.ble.rxLength = _rxLength;
            sysData.ble.newData = true;
            _dataReceived = false;        // フラグをクリア
        }
    }

    // updateOutput()は出力フェーズで呼ばれる
    void updateOutput(SystemData& sysData) override {
        if (sysData.ble.sendRequest) {
            // SystemDataの送信データをBLEで送信
            // ...
            sysData.ble.sendRequest = false;
        }
    }
};
\end{lstlisting}

\begin{notebox}[volatileフラグ方式を使う理由]
\texttt{volatile}は「この変数は外部から変更される可能性がある」とコンパイラに伝えるキーワードである。
これがないと、コンパイラの最適化によりフラグの変更が検出されない場合がある。\\
\\
\texttt{bool}型フラグ1個の読み書きは原子的であるため、
「バッファへの書き込み完了後にフラグを立てる」→「フラグ確認後にバッファを読む」の順序で
安全にデータを受け渡しできる。mutexは不要である。
\end{notebox}

\begin{cautionbox}[SystemDataへの直接アクセス禁止]
以下のコードは\textbf{禁止}である。BLEコールバックや割り込みハンドラから
\texttt{SystemData}を直接書き換えると、3フェーズループ実行中にデータが変わり不整合が発生する。
\begin{lstlisting}[style=cppstyle]
// NG: コールバックからSystemDataに直接書き込み
void onBleReceive(SystemData& data, uint8_t* payload) {
    data.ble.command = payload[0];  // 禁止
}
\end{lstlisting}
\end{cautionbox}

\subsection{内部バッファパターン}

カメラ等の重い処理を持つモジュールでは、メインループで取得したデータを
クラス内バッファに保存し、\texttt{updateInput()}呼び出し時に\texttt{SystemData}へコピーする。
これにより、\texttt{updateInput()}自体は軽量な処理に留まる。

\begin{lstlisting}[style=cppstyle, caption={内部バッファパターンの実装例}]
// CameraModule.h
struct CameraData {
    uint8_t* frame = nullptr;    // フレームデータポインタ
    size_t frameSize = 0;        // フレームサイズ
    bool newFrame = false;       // 新規フレームフラグ
    bool hasError = false;
};

class CameraModule : public IModule {
private:
    CameraConfig _config;
    // 内部バッファ
    uint8_t* _buffer = nullptr;
    size_t _bufferSize = 0;
    volatile bool _bufferReady = false;

public:
    CameraModule(const CameraConfig& config)
        : _config(config) {}

    bool init() override {
        // カメラハードウェア初期化
        // ...
        return true;
    }

    // メインループから直接呼ぶ（重い処理）
    void capture() {
        // カメラからフレーム取得（時間がかかる）
        camera_fb_t* fb = esp_camera_fb_get();
        if (fb) {
            _buffer = fb->buf;
            _bufferSize = fb->len;
            _bufferReady = true;
        }
    }

    // updateInput()は軽量：バッファからSystemDataへコピーするだけ
    void updateInput(SystemData& data) override {
        if (_bufferReady) {
            data.camera.frame = _buffer;
            data.camera.frameSize = _bufferSize;
            data.camera.newFrame = true;
            _bufferReady = false;
        }
    }
};
\end{lstlisting}

\begin{funcblock}{内部バッファパターンの流れ}
\begin{enumerate}
    \item \texttt{loop()}のメイン処理で\texttt{capture()}を呼び、重い処理を実行
    \item 処理完了時にクラス内バッファにデータを保存し、\texttt{\_bufferReady = true}にセット
    \item タイマー割り込みの3フェーズループで\texttt{updateInput()}が呼ばれる
    \item \texttt{updateInput()}はバッファから\texttt{SystemData}へコピーするだけ（軽量）
\end{enumerate}
\end{funcblock}

\subsection{モジュール実装テンプレート}

新規モジュールを追加する際の標準テンプレート。入力専用・出力専用・入出力の3パターンを示す。

\begin{lstlisting}[style=cppstyle, caption={入力モジュールテンプレート}]
// SensorModule.h
#pragma once
#include "IModule.h"
#include "ModuleTimer.h"

struct SensorConfig {
    uint8_t inputPin;
    unsigned long interval;   // 読み取り間隔(ms)
};

struct SensorData {
    int value = 0;
    bool isValid = false;
};

class SensorModule : public IModule {
private:
    SensorConfig _config;
    ModuleTimer _readTimer;
public:
    SensorModule(const SensorConfig& config)
        : _config(config) {}

    bool init() override {
        pinMode(_config.inputPin, INPUT);
        _readTimer.setTime();
        return true;
    }

    void updateInput(SystemData& data) override {
        if (_readTimer.getNowTime() >= _config.interval) {
            _readTimer.setTime();
            data.sensor.value = analogRead(_config.inputPin);
            data.sensor.isValid = true;
        }
    }
};
\end{lstlisting}

\begin{lstlisting}[style=cppstyle, caption={出力モジュールテンプレート}]
// ActuatorModule.h
#pragma once
#include "IModule.h"

struct ActuatorConfig {
    uint8_t outputPin;
};

struct ActuatorData {
    int power = 0;
};

class ActuatorModule : public IModule {
private:
    ActuatorConfig _config;
public:
    ActuatorModule(const ActuatorConfig& config)
        : _config(config) {}

    bool init() override {
        pinMode(_config.outputPin, OUTPUT);
        return true;
    }

    void updateOutput(SystemData& data) override {
        analogWrite(_config.outputPin, data.actuator.power);
    }
};
\end{lstlisting}
