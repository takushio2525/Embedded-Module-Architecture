% システムアーキテクチャ
\section{システムアーキテクチャ}

\subsection{全体構造}

本アーキテクチャは、以下のコア要素で構成される：

\begin{enumerate}
    \item \textbf{IModule}（インターフェース） — モジュールのライフサイクルを定義。プロジェクト非依存
    \item \textbf{ModuleTimer}（ユーティリティクラス） — ノンブロッキングなタイマー制御。プロジェクト非依存
    \item \textbf{SystemData}（共有データ構造体） — モジュール間のデータ連携。プロジェクト固有
    \item \textbf{モジュールConfig}（設定構造体） — 各モジュールのピンアサインや設定を構造体で定義。プロジェクト固有
\end{enumerate}

\begin{notebox}[コア層とプロジェクト層の分離]
\texttt{IModule}と\texttt{ModuleTimer}は\textbf{コア層}に属し、プロジェクト固有の型を一切知らない。\\
\texttt{SystemData}とモジュールConfigは\textbf{プロジェクト層}に属し、
前者は使用するモジュールのサブ構造体を集約し、後者はハードウェア構成（ピンアサイン等）を定義する。\\
\texttt{IModule}は\texttt{SystemData}を前方宣言で参照するため、コア層の完全な独立性を実現している。
\end{notebox}

\subsection{実行フロー}

プログラムの実行フローは、ゲームループパターンに基づき、3フェーズモデルで構成される：

\begin{lstlisting}[style=cppstyle, caption={3フェーズモデルによる実行フロー}]
// main.cpp
#include "IModule.h"
#include "ProjectConfig.h"  // SystemData + Config定義
#include "TempSensorModule.h"
#include "SwitchModule.h"
#include "LedModule.h"
#include "MotorModule.h"

// システムデータ
SystemData systemData;

// 入力モジュール（Config構造体を渡して生成）
TempSensorModule tempModule(TEMP_CONFIG);
SwitchModule swModule(SW_CONFIG);

// 出力モジュール
LedModule ledModule(LED_CONFIG);
MotorModule motorModule(MOTOR_CONFIG);

// 入力モジュール配列
IModule* inputModules[] = {
    &tempModule,
    &swModule,
};
const int INPUT_COUNT = sizeof(inputModules) / sizeof(inputModules[0]);

// 出力モジュール配列
IModule* outputModules[] = {
    &ledModule,
    &motorModule,
};
const int OUTPUT_COUNT = sizeof(outputModules) / sizeof(outputModules[0]);

// ロジックフェーズ（標準は関数として実装）
void applyPattern(SystemData& data) {
    // 入力SystemDataを読み、出力SystemDataを書き換える
    data.led.state = data.sw.isPressed;
    data.motor.speed = (data.temp.temperature > 30.0) ? 255 : 0;
}

void setup() {
    Serial.begin(115200);

    // 全モジュールを初期化（両配列に含まれるモジュールの重複initを避ける）
    IModule* allModules[] = {
        &tempModule, &swModule,
        &ledModule, &motorModule,
    };
    const int ALL_COUNT = sizeof(allModules) / sizeof(allModules[0]);

    const int MAX_RETRY = 3;
    for (int i = 0; i < ALL_COUNT; i++) {
        bool success = false;
        for (int r = 0; r < MAX_RETRY; r++) {
            if (allModules[i]->init()) { success = true; break; }
            delay(100);
        }
        if (!success) {
            Serial.printf("[System] Module %d: init failed, disabled\n", i);
            allModules[i]->enabled = false;
        }
    }
}

void loop() {
    // 1. 入力フェーズ：センサー値をSystemDataに書き込む
    for (int i = 0; i < INPUT_COUNT; i++) {
        if (inputModules[i]->enabled) {
            inputModules[i]->updateInput(systemData);
        }
    }
    // 2. ロジックフェーズ：入力→出力のSystemData変換
    applyPattern(systemData);
    // 3. 出力フェーズ：SystemDataの値に従いハードウェア制御
    for (int i = 0; i < OUTPUT_COUNT; i++) {
        if (outputModules[i]->enabled) {
            outputModules[i]->updateOutput(systemData);
        }
    }
}
\end{lstlisting}

\subsection{設計パターン}

\subsubsection{モジュール配列パターン}
\texttt{inputModules[]}と\texttt{outputModules[]}の2つの配列でモジュールを管理する。
配列の順序がそのまま各フェーズ内の実行順序となる。
これにより：
\begin{itemize}
    \item 入力と出力の処理順序が明示的かつ決定論的
    \item 入力→ロジック→出力のデータの流れが保証される
    \item 1ループにつき各モジュール1回の更新が保証される
    \item 各モジュールが独立した「道具」として機能する
\end{itemize}

\begin{notebox}[重要：処理順序]
各フェーズ内での配列順序はデータフローの順序を意味する。\\
例：温度センサーの値を補正センサーで校正する場合、温度センサーを補正センサーより前に配置する。\\
ただし、入力と出力のクロス参照（出力モジュールが入力のSystemDataを読む等）は行わない。
\end{notebox}

\subsubsection{モジュールのファイル構成}

各モジュールは\textbf{ヘッダー（.h）と実装（.cpp）に分離}する。
これは\texttt{SystemData}の循環依存を回避するための必須ルールである。

\begin{cautionbox}[循環依存の問題]
モジュールクラスは\texttt{IModule}を継承するため\texttt{SystemData}の型が必要となる。
しかし\texttt{SystemData}は各モジュールの\texttt{Data}構造体を含むため、モジュールヘッダーをインクルードする。\\
ヘッダー内にメソッドをインライン実装すると、
\texttt{LedModule.h} → \texttt{SystemData} → \texttt{LedModule.h}
の循環が発生してコンパイルエラーとなる。
\end{cautionbox}

\texttt{SystemData}を\texttt{include/SystemData.h}に分離し、
モジュールの\texttt{.cpp}では\texttt{SystemData.h}のみをインクルードすることで循環依存を回避する。
\texttt{ProjectConfig.h}はConfigインスタンスの定義のみを担当し、\texttt{main.cpp}だけが参照する。

\begin{funcblock}{モジュールファイル構成のルール}
\begin{enumerate}
    \item \textbf{ヘッダー（.h）}: Config構造体、Data構造体、\texttt{SystemData}の前方宣言、クラス宣言（メソッドはプロトタイプのみ）
    \item \textbf{実装（.cpp）}: \texttt{SystemData.h}をインクルードし、メソッドを実装（\texttt{ProjectConfig.h}はインクルードしない）
    \item ヘッダーでは\texttt{<Arduino.h>}と\texttt{"IModule.h"}をインクルードする
    \item ヘッダーでは\texttt{SystemData.h}をインクルードしない（循環依存を防ぐため）
\end{enumerate}
\end{funcblock}

\begin{lstlisting}[style=cppstyle, caption={モジュールのファイル構成例}]
// --- LedModule.h（ヘッダー：宣言のみ） ---
#pragma once
#include <Arduino.h>
#include "IModule.h"

struct LedConfig {
    uint8_t ledPin;
};

struct LedData {
    bool state = false;
    uint8_t brightness = 0;
};

struct SystemData;  // 前方宣言（実体はSystemData.hで定義）

class LedModule : public IModule {
private:
    LedConfig _config;
public:
    LedModule(const LedConfig& config);
    bool init() override;
    void updateOutput(SystemData& data) override;
};

// --- LedModule.cpp（実装：SystemData.hで完全な定義を取得） ---
#include "LedModule.h"
#include "SystemData.h"

LedModule::LedModule(const LedConfig& config) : _config(config) {}

bool LedModule::init() {
    pinMode(_config.ledPin, OUTPUT);
    Serial.println("[Led] init OK");
    return true;
}

void LedModule::updateOutput(SystemData& data) {
    digitalWrite(_config.ledPin, data.led.state ? HIGH : LOW);
}
\end{lstlisting}

\subsubsection{SystemDataパターン}
各モジュールは固有のサブ構造体を定義し、それらをプロジェクト固有の\texttt{SystemData}構造体に集約する。

\begin{lstlisting}[style=cppstyle, caption={SystemDataパターンの構造}]
// 各モジュールが自身のデータ構造体を定義（ヘッダー内）
// メンバには必ずデフォルト値を明示する
// LedModule.h 内
struct LedData {
    bool state = false;
    uint8_t brightness = 0;
};

// TempSensorModule.h 内
struct TempData {
    float temperature = -999.0f;  // 無効値
    bool isValid = false;         // まだ読み取っていない
    bool hasError = false;
};

// SystemData.h 内: 各モジュールのData構造体を集約
struct SystemData {
    LedData led;
    TempData temp;
    // 使用するモジュールの構造体を追加するだけ
};
\end{lstlisting}

\subsubsection{モジュールConfigパターン}
各モジュールは、自身に必要なピンアサインや設定を\texttt{Config}構造体として定義する。
モジュールのコンストラクタではピン番号を個別に受け取るのではなく、Config構造体を受け取ることで、
ハードウェア構成を明示的かつ安全に管理する。

\begin{funcblock}{Configパターンの利点}
\begin{itemize}
    \item コンストラクタの引数順序を覚える必要がない（構造体のメンバ名で明示的に指定）
    \item 全モジュールのピンアサインが\texttt{ProjectConfig.h}に集約され、一覧性が高い
    \item 回路設計担当者は\texttt{ProjectConfig.h}のConfig定義セクションだけを見れば回路設計が可能
    \item 将来的にピン以外の設定（I2Cアドレス、動作モード等）もConfig構造体に追加して拡張可能
\end{itemize}
\end{funcblock}

\begin{funcblock}{ピン定義のルール — 構造体リテラルへの直書き}
モジュール固有のピン番号は\texttt{constexpr}単体定数にせず、Configインスタンスのリテラルに\textbf{直書き}する。
外部からピンを参照する場合は\texttt{SERVO\_CONFIG.pin}のようにConfig構造体経由でアクセスする。
これにより「どのモジュールのピンか」が常に明確になり、中間変数による二重管理を排除する。

\medskip
\textbf{例外:} 共有バスピン（\texttt{SPI\_MOSI\_PIN}等）は特定モジュールに属さない共有インフラのため、
\texttt{constexpr}単体定数として定義してよい。\texttt{main.cpp}の\texttt{bus.begin()}で直接使用するため。

\begin{lstlisting}[style=cppstyle, caption={ピン定義の正しい書き方}]
// ===== 共有バスピン: 単体定数OK =====
constexpr int SPI_MOSI_PIN = 38;
constexpr int SPI_SCK_PIN  = 39;

// ===== モジュール固有ピン: Configリテラルに直書き =====
const ServoConfig SERVO_CONFIG = {
    .pin = 1,   // 直書き（constexpr int SERVO_PIN = 1; は不要）
};
const SdConfig SD_CONFIG = {
    .csPin = 21, // 直書き（constexpr int SD_CS_PIN = 21; は不要）
};

// 外部から参照する場合は構造体経由
Serial.printf("Servo pin: %d\n", SERVO_CONFIG.pin);
\end{lstlisting}
\end{funcblock}

\begin{lstlisting}[style=cppstyle, caption={モジュールConfigパターンの例}]
// LedModule.h 内 — Config構造体とクラス宣言
struct LedConfig {
    uint8_t ledPin;      // LED出力ピン
};

struct SystemData;  // 前方宣言

class LedModule : public IModule {
private:
    LedConfig _config;
public:
    LedModule(const LedConfig& config);  // 実装は.cppで
    bool init() override;
    void updateOutput(SystemData& data) override;
};

// ProjectConfig.h 内 — プロジェクトのピンアサインを集約
const LedConfig LED_CONFIG = { .ledPin = 13 };

// main.cpp — Config定数を渡してインスタンス生成
LedModule ledModule(LED_CONFIG);
\end{lstlisting}

\subsubsection{通信バスパターン}
SPI・I2C等の通信バスは複数モジュールで共有されるリソースであり、
個々のモジュールが所有・初期化すべきではない。
バスインスタンスを\texttt{main.cpp}で生成し、\textbf{コンストラクタ引数}でモジュールに渡す。

\begin{funcblock}{通信バスを共有リソースとして管理する理由}
\begin{itemize}
    \item 複数モジュールが同じバスを使う場合、各モジュールが\texttt{begin()}すると競合・二重初期化が発生する
    \item バスのピンアサインやインスタンスがモジュール内にハードコードされると、モジュールの再利用性が損なわれる
    \item バスの生成と初期化を\texttt{main.cpp}に集約することで、共有リソースの管理が明確になる
\end{itemize}
\end{funcblock}

\begin{funcblock}{バスポインタをConfig構造体に含めない理由}
Config構造体のインスタンスは\texttt{ProjectConfig.h}で定義する。
もしConfig構造体にバスポインタを含めると、\texttt{ProjectConfig.h}より前にバスインスタンスを
グローバルスコープで生成し\texttt{extern}宣言する必要があり、インクルード順序の制約が生じる。\\
バスの紐付けをコンストラクタ引数に分離することで、
Config構造体はピンアサイン等の純粋な設定値のみを保持し、
\texttt{ProjectConfig.h}はバスインスタンスの存在を知る必要がなくなる。
\end{funcblock}

\paragraph{SPIバスの例}

\begin{lstlisting}[style=cppstyle, caption={SPIバスを使用するモジュールの実装例}]
// SpiSensorModule.h
#pragma once
#include <SPI.h>
#include "IModule.h"
#include "ModuleTimer.h"

// --- Config構造体（デバイス固有設定のみ、バスポインタは含めない） ---
struct SpiSensorConfig {
    uint8_t csPin;            // チップセレクトピン
    uint32_t clockSpeed;      // SPIクロック速度(Hz)
    unsigned long interval;   // 読み取り間隔(ms)
};

// --- Data構造体 ---
struct SpiSensorData {
    uint16_t rawValue = 0;
    bool hasError = false;
};

class SpiSensorModule : public IModule {
private:
    SpiSensorConfig _config;
    SPIClass* _spi;           // コンストラクタで受け取るバスポインタ
    ModuleTimer _readTimer;
public:
    // Config + バスポインタをコンストラクタで受け取る
    SpiSensorModule(const SpiSensorConfig& config, SPIClass* spi)
        : _config(config), _spi(spi) {}

    bool init() override {
        // CSピンの設定のみ行う（bus.begin()は呼ばない）
        pinMode(_config.csPin, OUTPUT);
        digitalWrite(_config.csPin, HIGH);
        _readTimer.setTime();
        return true;
    }

    void updateInput(SystemData& data) override {
        if (_readTimer.getNowTime() >= _config.interval) {
            _readTimer.setTime();
            // beginTransaction/endTransactionで排他制御
            _spi->beginTransaction(
                SPISettings(_config.clockSpeed, MSBFIRST, SPI_MODE0)
            );
            digitalWrite(_config.csPin, LOW);
            data.spiSensor.rawValue = _spi->transfer16(0x00);
            digitalWrite(_config.csPin, HIGH);
            _spi->endTransaction();
        }
    }
};
\end{lstlisting}

\paragraph{I2Cバスの例}

\begin{lstlisting}[style=cppstyle, caption={I2Cバスを使用するモジュールの実装例}]
// I2cSensorModule.h
#pragma once
#include <Wire.h>
#include "IModule.h"
#include "ModuleTimer.h"

// --- Config構造体（デバイス固有設定のみ、バスポインタは含めない） ---
struct I2cSensorConfig {
    uint8_t address;          // I2Cデバイスアドレス
    unsigned long interval;   // 読み取り間隔(ms)
};

// --- Data構造体 ---
struct I2cSensorData {
    float value = 0.0f;
    bool hasError = false;
};

class I2cSensorModule : public IModule {
private:
    I2cSensorConfig _config;
    TwoWire* _wire;           // コンストラクタで受け取るバスポインタ
    ModuleTimer _readTimer;
public:
    // Config + バスポインタをコンストラクタで受け取る
    I2cSensorModule(const I2cSensorConfig& config, TwoWire* wire)
        : _config(config), _wire(wire) {}

    bool init() override {
        // バスの初期化は行わない（main.cppで実施済み）
        _readTimer.setTime();
        return true;
    }

    void updateInput(SystemData& data) override {
        if (_readTimer.getNowTime() >= _config.interval) {
            _readTimer.setTime();
            _wire->beginTransmission(_config.address);
            _wire->write(0x00);  // レジスタアドレス
            if (_wire->endTransmission() != 0) {
                data.i2cSensor.hasError = true;
                return;
            }
            _wire->requestFrom(_config.address, (uint8_t)2);
            if (_wire->available() >= 2) {
                uint8_t high = _wire->read();
                uint8_t low = _wire->read();
                data.i2cSensor.value =
                    (float)((high << 8) | low) / 100.0;
                data.i2cSensor.hasError = false;
            }
        }
    }
};
\end{lstlisting}

\paragraph{main.cppでのバス生成・初期化}

\begin{lstlisting}[style=cppstyle, caption={main.cppでの通信バス生成と初期化}]
// main.cpp
#include <SPI.h>
#include <Wire.h>
#include "ProjectConfig.h"
#include "SpiSensorModule.h"
#include "I2cSensorModule.h"

// バスインスタンスの生成（グローバルスコープ）
SPIClass spi(HSPI);
TwoWire wire(0);

// モジュール生成（Config + バスポインタをコンストラクタで渡す）
SpiSensorModule spiModule(SPI_SENSOR_CONFIG, &spi);
I2cSensorModule i2cModule(I2C_SENSOR_CONFIG, &wire);

void setup() {
    Serial.begin(115200);

    // バス初期化（全モジュールのinit()より前）
    spi.begin(SCK_PIN, MISO_PIN, MOSI_PIN);
    wire.begin(SDA_PIN, SCL_PIN);

    // モジュール初期化
    spiModule.init();
    i2cModule.init();
}
\end{lstlisting}

\begin{notebox}[通信バスパターンのルール]
\begin{enumerate}
    \item バスインスタンスは\texttt{main.cpp}のグローバルスコープで生成する
    \item \texttt{bus.begin()}は\texttt{setup()}内で、全モジュールの\texttt{init()}より前に呼ぶ
    \item バスポインタはConfig構造体に含めず、\textbf{コンストラクタの引数}でモジュールに渡す
    \item モジュールはバスポインタをメンバ変数（\texttt{\_spi}, \texttt{\_wire}等）として保持する
    \item モジュールの\texttt{init()}で\texttt{bus.begin()}を呼ばない（CSピン設定等のみ行う）
    \item SPI使用時は\texttt{beginTransaction()}/\texttt{endTransaction()}で排他制御する
\end{enumerate}
\end{notebox}

\subsubsection{Data構造体のデフォルト値ルール}

各\texttt{\{Module\}Data}構造体のメンバには、必ずデフォルト値を明示する。
これにより、\texttt{updateInput()}/\texttt{updateOutput()}が1度も呼ばれていない状態でも安全にデータを参照できる。

\begin{funcblock}{デフォルト値の指針}
\begin{itemize}
    \item \texttt{bool}型のフラグ: \texttt{false}で初期化（\texttt{isValid = false}, \texttt{hasError = false}）
    \item 数値型: 正常値と区別できる無効値で初期化（例: \texttt{temperature = -999.0f}）
    \item ポインタ: \texttt{nullptr}で初期化
\end{itemize}
\end{funcblock}

\subsubsection{データアクセスルール}

\texttt{updateInput()}/\texttt{updateOutput()}内では、\textbf{自モジュールのサブ構造体のみ}にアクセスする。
他モジュールのサブ構造体への読み書きは\textbf{禁止}とする。
モジュール間のデータ連携が必要な場合は、必ず\textbf{ロジックフェーズ}を経由する。

\begin{lstlisting}[style=cppstyle, caption={データアクセスルールの例}]
// OK: 自モジュールのサブ構造体のみアクセス
void TempSensorModule::updateInput(SystemData& data) {
    data.temp.temperature = analogRead(_config.tempPin);
    data.temp.isValid = true;
}

// NG: 他モジュールのサブ構造体にアクセスしている
void TempSensorModule::updateInput(SystemData& data) {
    data.temp.temperature = analogRead(_config.tempPin);
    data.led.state = true;  // 禁止：LEDはロジックフェーズで制御する
}
\end{lstlisting}

\subsubsection{モジュール統合の原則}

\textbf{モジュール間の依存関係は存在しないものとして設計する。}
各モジュールは他のモジュールの存在を知らず、独立して動作する。
依存関係がある場合は、それらを\textbf{1つの大きなモジュールに統合}する。

例：カメラモジュールがSDカードを必要とする場合、「カメラ＋SDカードモジュール」として1つにまとめる。

\subsubsection{enabledフラグによる動的管理}

\texttt{IModule}の\texttt{enabled}フラグにより、実行時のモジュール有効・無効切り替えを実現する。

\begin{itemize}
    \item \texttt{enabled = true}（デフォルト）: 通常通り\texttt{updateInput()}/\texttt{updateOutput()}が呼ばれる
    \item \texttt{enabled = false}: ループ内で\texttt{updateInput()}/\texttt{updateOutput()}がスキップされる
    \item BLE等の外部通信からモジュールの\texttt{enabled}を切り替えることで、動的な制御が可能
\end{itemize}

\begin{lstlisting}[style=cppstyle, caption={enabledフラグの使用例}]
void loop() {
    for (int i = 0; i < INPUT_COUNT; i++) {
        if (inputModules[i]->enabled) {
            inputModules[i]->updateInput(systemData);
        }
    }
    applyPattern(systemData);
    for (int i = 0; i < OUTPUT_COUNT; i++) {
        if (outputModules[i]->enabled) {
            outputModules[i]->updateOutput(systemData);
        }
    }
}
\end{lstlisting}

\subsubsection{エラーハンドリングパターン}

\paragraph{init()失敗時}
規定回数（例：3回）までリトライし、上限を超えた場合は\texttt{enabled = false}に設定してモジュールを無効化する。
シリアルモニタにエラーログを出力する。

\begin{lstlisting}[style=cppstyle, caption={init()リトライパターンの実装例}]
// setup() 内
const int MAX_RETRY = 3;
for (int i = 0; i < INPUT_COUNT; i++) {
    bool success = false;
    for (int retry = 0; retry < MAX_RETRY; retry++) {
        if (inputModules[i]->init()) {
            success = true;
            break;
        }
        delay(100);  // init()のリトライ間隔（setup内のみdelay許容）
    }
    if (!success) {
        Serial.printf("Input Module %d: init failed, disabled\n", i);
        inputModules[i]->enabled = false;
    }
}
\end{lstlisting}

\paragraph{updateInput()/updateOutput()エラー時}
Data構造体内のエラーフラグ（\texttt{hasError}等）を\texttt{true}に設定する。
モジュールは次の\texttt{updateInput()}/\texttt{updateOutput()}サイクルで再試行し、成功時にモジュール自身がエラーフラグを\texttt{false}にクリアする。

\begin{lstlisting}[style=cppstyle, caption={updateInput()エラーフラグの運用例}]
void I2cSensorModule::updateInput(SystemData& data) {
    if (_readTimer.getNowTime() >= _config.interval) {
        _readTimer.setTime();
        if (readSensor()) {
            data.i2cSensor.value = _lastValue;
            data.i2cSensor.hasError = false;  // 成功時にクリア
        } else {
            data.i2cSensor.hasError = true;   // 失敗時にセット
        }
    }
}
\end{lstlisting}

\subsubsection{init失敗後の復帰パターン}

\texttt{init()}失敗で\texttt{enabled = false}にしたモジュールは、
ロジックフェーズで定期的に\texttt{init()}を再試行し、成功時に\texttt{enabled = true}に復帰させる。

\begin{lstlisting}[style=cppstyle, caption={ロジックフェーズでのinit復帰パターン}]
// main.cpp
ModuleTimer reinitTimer;

void tryReinitModules() {
    if (reinitTimer.getNowTime() < 5000) return;  // 5秒間隔
    reinitTimer.setTime();

    for (int i = 0; i < INPUT_COUNT; i++) {
        if (!inputModules[i]->enabled) {
            if (inputModules[i]->init()) {
                Serial.printf("[System] Input Module %d: reinit OK\n", i);
                inputModules[i]->enabled = true;
            }
        }
    }
    for (int i = 0; i < OUTPUT_COUNT; i++) {
        if (!outputModules[i]->enabled) {
            if (outputModules[i]->init()) {
                Serial.printf("[System] Output Module %d: reinit OK\n", i);
                outputModules[i]->enabled = true;
            }
        }
    }
}

void loop() {
    // 1. 入力フェーズ ...
    // 2. ロジックフェーズ
    applyPattern(systemData);
    tryReinitModules();  // 無効モジュールの復帰チェック
    // 3. 出力フェーズ ...
}
\end{lstlisting}

\subsubsection{ログ出力フォーマット}

シリアルモニタへのログ出力は、以下のフォーマットに統一する。

\begin{funcblock}{ログフォーマット}
\texttt{[モジュール名] メッセージ}\\
例: \texttt{[Camera] init failed: timeout}\\
例: \texttt{[Temp] センサー読み取りエラー}\\
例: \texttt{[System] Input Module 0: reinit OK}
\end{funcblock}

デバッグ用のログは\texttt{\#ifdef DEBUG}等で切り替え可能にし、
リリースビルドでは不要なログを無効化できるようにする。

\subsubsection{Configの実行時変更}

Config構造体は\textbf{初期値}として扱う。
コンストラクタでコピーされたメンバ変数\texttt{\_config}は\texttt{const}ではないため、
実行時に変更が必要な場合は\textbf{モジュールインスタンス内の\_configメンバを直接変更}する。

\begin{cautionbox}[Config変更のタイミング]
Configの変更（セッター呼び出し）は\textbf{ロジックフェーズでのみ}行う。
BLE等の外部からの変更要求は\texttt{SystemData}経由でロジックフェーズに渡し、
ロジックフェーズ内でセッターを呼び出す。
これにより、\texttt{updateInput()}/\texttt{updateOutput()}実行中にConfigが変わるデータ不整合を防ぐ。
\end{cautionbox}

\begin{lstlisting}[style=cppstyle, caption={Config値の実行時変更例}]
class TempSensorModule : public IModule {
private:
    TempSensorConfig _config;  // constではない→変更可能
public:
    TempSensorModule(const TempSensorConfig& config)
        : _config(config) {}

    // 実行時にintervalを変更するセッター
    void setInterval(unsigned long interval) {
        _config.interval = interval;
    }

    // updateInput()は変更後の値を使用する
    void updateInput(SystemData& data) override {
        if (_readTimer.getNowTime() >= _config.interval) {
            // ...
        }
    }
};

// ロジックフェーズ等から変更する例
void applyPattern(SystemData& data) {
    if (data.ble.newIntervalReceived) {
        tempModule.setInterval(data.ble.newInterval);
    }
}
\end{lstlisting}
